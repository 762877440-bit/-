# 项目需求文档 (Requirement Specification)

## 1. 项目背景与目标
### 1.1 背景
当前通用 AI 助手多依赖于云端服务，存在隐私顾虑、网络依赖和对话记忆不连贯等问题。本项目旨在构建一个**本地优先、具备长期记忆和自主规划能力**的终端 AI 助手，为普通消费者提供专业、可控的智能对话体验。

### 1.2 目标
- 打造一个纯文本交互的终端 AI 助手，核心模型为 Qwen3.4-4b。
- 实现跨会话的长期记忆，使助手能记住用户的偏好和关键信息。
- 赋予助手实时网页搜索和自主任务规划能力，能够应对复杂查询。
- 保证流畅的流式输出和专业的中文语气。

## 2. 用户角色
- **终端用户**：普通消费者，无需技术背景，通过自然语言与助手对话。
- **开发者（可选）**：后续可能对助手进行配置或扩展功能。

## 3. 功能需求

### 3.1 对话交互
| 需求ID | 需求描述 | 优先级 |
|--------|----------|--------|
| REQ-CONV-01 | 系统应提供命令行交互界面，用户输入后获得文本回复。 | P0 |
| REQ-CONV-02 | 系统必须支持连续多轮对话，维护当前会话的上下文（至少 8k tokens）。 | P0 |
| REQ-CONV-03 | 系统必须以流式（逐词/逐字）方式输出回复，减少等待感。 | P0 |
| REQ-CONV-04 | 系统应默认使用中文回复，并能正确处理中英文混合输入。 | P0 |
| REQ-CONV-05 | 系统回复应采用专业、客观、有条理的语气，避免口语化或情绪化表达。 | P0 |
| REQ-CONV-06 | 用户可输入特殊命令：`exit` 退出程序，`/clear` 清空当前对话上下文，`/memory` 查看已存储的长期记忆，`/search on/off` 临时开关联网搜索功能。 | P1 |

### 3.2 长期记忆
| 需求ID | 需求描述 | 优先级 |
|--------|----------|--------|
| REQ-MEM-01 | 系统应具备跨会话的长期记忆能力：关闭并重新启动程序后，仍能回忆之前对话中的事实性信息（如用户名、职业、兴趣爱好等）。 | P0 |
| REQ-MEM-02 | 系统应自动判断哪些信息值得长期存储，无需用户手动标记。 | P0 |
| REQ-MEM-03 | 系统应在每轮对话开始时，自动检索与当前问题相关的历史记忆，并注入上下文。 | P0 |
| REQ-MEM-04 | 当用户提供与已存储记忆相矛盾的新信息时，系统应能更新或覆盖旧记忆。 | P1 |
| REQ-MEM-05 | 用户应能够通过 `/memory delete <关键字>` 命令删除特定记忆。 | P2 |

### 3.3 实时网页搜索
| 需求ID | 需求描述 | 优先级 |
|--------|----------|--------|
| REQ-SEARCH-01 | 系统必须集成网页搜索能力，能够根据用户问题自主判断是否需要联网获取最新信息。 | P0 |
| REQ-SEARCH-02 | 系统应调用公开的搜索 API（如 DuckDuckGo）获取网页标题、摘要和 URL。 | P0 |
| REQ-SEARCH-03 | 系统应将搜索结果作为上下文的一部分，辅助模型生成更准确的回答。 | P0 |
| REQ-SEARCH-04 | 回复中应注明主要信息来源（至少包含网站名称）。 | P1 |
| REQ-SEARCH-05 | 若搜索请求超时（5 秒）或失败，系统应降级为不依赖搜索的普通回答，并向用户提示“当前无法联网”。 | P1 |

### 3.4 自主规划与执行
| 需求ID | 需求描述 | 优先级 |
|--------|----------|--------|
| REQ-AGENT-01 | 对于需要多步骤解决的问题（如“对比今日和昨日比特币价格并给出投资建议”），系统应能自主将任务分解为若干子步骤（例如：搜索今日价格 → 搜索昨日价格 → 计算变化 → 生成建议）。 | P0 |
| REQ-AGENT-02 | 系统应能自主调用合适的工具（搜索、记忆读写、计算器等）并按规划顺序执行。 | P0 |
| REQ-AGENT-03 | 当工具调用失败（如网络错误、搜索无结果）时，系统应尝试替代方案（如更换搜索词）或向用户请求进一步说明。 | P1 |
| REQ-AGENT-04 | 系统应提供可选的“详细模式”（verbose），显示决策过程和工具调用链，便于高级用户理解 Agent 行为。 | P2 |

### 3.5 终端体验增强
| 需求ID | 需求描述 | 优先级 |
|--------|----------|--------|
| REQ-UI-01 | 用户输入提示符应为 `You > `，助手回复提示符应为 `Agent > `。 | P0 |
| REQ-UI-02 | 不同信息类型应使用不同颜色输出：用户输入（绿色）、助手回复（白色）、思考/调用过程（灰色）、错误信息（红色）。 | P1 |
| REQ-UI-03 | 支持使用上下箭头键浏览历史输入（类似 shell 的 history）。 | P2 |

## 4. 非功能需求

| 需求ID | 类别 | 需求描述 | 优先级 |
|--------|------|----------|--------|
| REQ-NF-01 | 性能 | 首 token 延迟 ≤ 2 秒（推荐配置下）。生成速度 ≥ 20 token/秒。 | P0 |
| REQ-NF-02 | 性能 | 记忆检索延迟 ≤ 500 毫秒。 | P0 |
| REQ-NF-03 | 可靠性 | 系统应能连续运行 8 小时无内存泄漏，内存占用增长不超过初始值的 20%。 | P1 |
| REQ-NF-04 | 可维护性 | 所有外部依赖（搜索 API、记忆存储、模型调用）应通过配置文件切换实现松耦合。 | P1 |
| REQ-NF-05 | 可测试性 | 核心逻辑（工作流决策、工具调用）应有单元测试覆盖率 ≥ 70%。 | P1 |
| REQ-NF-06 | 安全性 | 用户对话数据和长期记忆仅存储在本地，不上传到任何远程服务器（搜索 API 请求本身除外）。 | P0 |
| REQ-NF-07 | 易用性 | 提供一键启动脚本或详细的环境配置文档，普通用户能在 15 分钟内完成安装和首次运行。 | P1 |

## 5. 约束条件
- 模型推理必须本地运行（Qwen3.4-4b），不得依赖云端大模型 API。
- 搜索功能必须可通过免费或无 API Key 的方式实现（如 DuckDuckGo HTML 抓取或官方免费接口），避免强制付费。
- 系统应能在没有 GPU 的环境下运行（降级为 CPU 推理，速度可能较慢，但功能完整）。
- 所有代码采用 Python 3.10+，并遵循 PEP 8 规范。

## 6. 验收标准
- 用户能够启动程序，输入“你好，我是张三”，关闭后再次启动，询问“我叫什么？”，系统正确回答“张三”。
- 用户询问“今天北京的天气如何”，系统能主动联网搜索并返回包含天气信息和来源的回答。
- 用户提出复杂任务“比较昨天和今天的黄金价格”，系统能依次执行两次搜索并给出对比结论。
- 所有特殊命令（`/clear`、`/memory`、`/search off` 等）功能正常。
- 流式输出效果明显，字符逐个出现。
